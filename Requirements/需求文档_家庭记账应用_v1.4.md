# 家庭记账应用需求文档 v1.4（多账本体验与同步修复）

## 1. 需求概述

- 一句话描述：修复 v1.3 多账本相关体验问题，补齐“新设备登录后账本自动可见可切换”、以及邀请码管理入口在部署环境不可用的问题。
- 目标用户：已经在 v1.3 使用“后端+同步”的家庭用户（单账号起步）。
- 背景与来源：来自 v1.3 测试报告中未通过项与体验反馈（邀请码管理页入口、账本同步、新手路径评分）。

## 2. 功能需求

### 2.1 多账本选择的体验边界（关键修复）

- `[ ]` 账户页“多账本选择”仅影响账户页统计展示
  - 账户页允许选择多个账本用于资产概览统计
  - 账户页的选择状态不应改变“当前账本”的全局状态
  - 账户页切换多账本后：
    - 首页仍展示“当前账本”的数据（或用户明确选择的账本）
    - “切换账本”功能不应失效

- `[ ]` 当前账本（全局）保持单选且可预期
  - 首页/新增/统计/报表等页面默认以“当前账本”作为数据来源
  - 若后续引入“跨账本聚合视图”，必须以显式入口与状态提示区分，且不覆盖当前账本

### 2.2 新设备登录后的同步初始化（关键修复）

- `[ ]` 新设备首次登录后自动拉取账本并进入可用状态
  - 场景：浏览器 B 首次打开（本地无数据）-> 登录现有账号
  - 期望：无需先“新建账本”，也能看到服务器已有账本列表，并可切换与使用
  - 若服务端存在账本：
    - 自动选择一个账本作为当前账本（默认：最近更新或列表第一个）
    - 自动加载该账本的分类/模板/标签与记录数据
  - 若服务端无账本：
    - 引导用户创建首个账本，并在创建后触发同步

### 2.3 账本同步的端到端可见性（补齐验收口径）

- `[ ]` 账本创建/切换在多设备间可同步与可恢复
  - 设备 A 新建账本 -> 设备 B 登录并同步后可看到并可切换
  - 设备 A 删除账本 -> 设备 B 同步后账本列表一致（删除策略沿用现有实现）
  - 账本相关的分类/模板/标签/记录数据与账本一致呈现

### 2.4 邀请码管理入口在部署环境可用（关键修复）

- `[ ]` 设置页“邀请码管理”入口可正确打开后端管理页
  - 在前后端不同域名、或被 PWA/Service Worker 接管路由的情况下，也能打开真实的后端页面
  - 管理页可正常调用后端接口（生成/查看/作废），且不被前端路由重写到首页

## 3. 验收标准（可观察、可复现）

- `1.` 账户页选择多个账本后，首页数据不受影响，仍按当前账本展示；切换账本功能可正常工作。
- `2.` 新浏览器（本地无数据）登录后，在一次自动同步或点击“立即同步”后，能看到服务端已有账本列表，无需新建账本即可切换使用。
- `3.` 在设备 A 新建账本后，设备 B 同步后可见该账本且可切换；切换后分类与记录展示正确。
- `4.` 设置页点击“邀请码管理”能打开后端管理页，不会跳回应用首页；在管理页可生成邀请码并刷新列表看到新增项。
- `5.` v1.3 已通过的核心能力保持不回退：登录态保持、CSRF、防抖退避、回收站删除/恢复/清空跨端一致。

## 4. 范围边界

### In（包含）

- 修复账户页多账本选择影响全局数据的问题
- 新设备登录后同步初始化与“自动选择当前账本”策略
- 补齐账本同步的端到端体验（账本可见、可切换、跨设备一致）
- 修复邀请码管理入口在部署环境不可用的问题

### Out（不包含）

- 多账号/家庭成员/复杂权限模型
- 复杂冲突合并（按字段合并/手动选择）
- 新增“找回密码/重置账号”的完整流程（如需可在后续版本单独立项）

## 5. 约束与假设

- 继续沿用 v1.3 技术栈：前端 React + TypeScript；后端 FastAPI + SQLite；公网通过 Cloudflare Tunnel。
- 继续沿用 v1.3 单账号模式：不增加第二个账号注册入口。

## 6. 风险与回滚

- 风险：
  - 多账本选择状态拆分后若处理不当，可能出现“页面数据源不一致”的问题
  - 新设备初始化逻辑若覆盖本地状态，可能导致用户误以为数据丢失（需要明确加载/同步状态）
- 回滚：
  - 可通过配置开关回退到“仅当前账本单选展示”（实现阶段定案）
  - 保留 v1.3 可运行版本与存档用于回退验证
