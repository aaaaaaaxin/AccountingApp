# 家庭记账应用需求文档 v1.3（后端与同步）

## 1. 需求概述
- 一句话描述：为现有纯前端记账应用新增后端服务，实现“简单登录 + 多设备数据同步”，并可部署到群晖 NAS 供公网访问。
- 目标用户：家庭用户（单账号起步，后续可扩展多账号/家庭成员）。
- 使用场景：
  - 手机记一笔 → 电脑/平板自动同步看到相同数据
  - 外出使用移动网络也可访问（公网）
  - NAS 上集中存储数据，便于备份与运维

## 2. 功能需求
### 2.1 核心功能
- `[ ]` 账号与登录（简单登录）
  - 支持注册、登录、退出
  - 登录态维持：支持刷新后保持登录（token 续期或重新登录策略需定案）
  - 账号数量（v1.3首发）：单账号
  - 注册方式（v1.3首发）：邀请注册（管理员生成邀请码）
  - 邀请码规则（v1.3首发）：一次性使用 + 7天有效期

- `[ ]` 数据域与隔离
  - 数据按账号隔离（v1.3若为单账号则等价于一个数据空间）
  - 数据对象范围（与前端保持一致）：
    - 账本（Ledger）
    - 分类（Category，含 parentId）
    - 记录（Transaction，含 tags、paymentMethod）
    - 模板（Template）
    - 标签（Tag）

- `[ ]` 同步机制（v1.3首发：自动后台同步）
  - 启动应用后自动拉取服务器最新数据并合并到本地
  - 本地写入（新增/编辑/删除）后自动后台推送到服务器
  - 网络异常时进入离线队列，网络恢复后自动补偿同步
  - 可观测性（v1.3首发）：显示同步状态 + 提供“立即同步”按钮兜底
  - 状态位置（v1.3首发）：设置页
  - 同步频率（v1.3首发）：事件触发 + 失败指数退避（启动/回到前台/本地变更后触发）

- `[ ]` 冲突处理（v1.3首发：最后写入覆盖）
  - 当同一条数据被多端修改产生冲突时，以“最后写入覆盖（LWW）”为准
  - 时间源（v1.3首发）：服务器接收时间

### 2.2 API 能力（对前端的最小接口集合）
- `[ ]` 认证接口
  - 注册/登录/刷新登录态/登出
  - 登录态（v1.3首发）：HttpOnly Cookie
- `[ ]` 邀请注册管理（v1.3首发：简易管理页）
  - 生成一次性邀请码（7天有效）
  - 查看邀请码状态（有效/已使用/已过期）
  - 作废未使用的邀请码
- `[ ]` 同步接口（v1.3首发：增量同步）
  - 按“变更日志（oplog）/版本号”拉取增量并上报增量
  - 删除采用“回收站”（软删除 + 可恢复）
  - 回收站保留（v1.3首发）：30天自动清理
  - 回收站清空（v1.3首发）：支持“清空回收站/彻底删除”，清空后不可恢复
  - 清空确认（v1.3首发）：清空回收站需要二次确认
    - 标题：清空回收站
    - 内容：将永久删除回收站内的所有数据，删除后不可恢复。确定继续吗？
    - 按钮：取消 / 彻底删除
  - 回收站范围（v1.3首发）：仅交易记录（Transaction）
  - oplog 归档（v1.3首发）：保留90天 + 定期截断（超期合并为基线并截断）

### 2.3 安全与合规（公网部署必备）
- `[ ]` 全站 HTTPS
- `[ ]` 基础防护
  - 密码安全存储（不可明文）
  - 限流/登录失败保护（避免暴力破解）
  - 基础安全头与 CORS 策略（仅允许前端域名）
- `[ ]` 数据备份策略（v1.3至少提供运维层面方案）
  - 容器数据落盘到 NAS 卷
  - 提供定期备份数据库文件/目录的方式（可脚本/群晖任务）
  - 备份承诺（v1.3首发）：提供备份方案，不提供应用内一键导入/导出

### 2.4 部署与运维（群晖 NAS）
- `[ ]` Docker 部署
  - 一键启动（docker compose 或同等方式）
  - 数据落盘可迁移
- `[ ]` 运维与资源约束（v1.3首发）
  - 日志：按天滚动 + 保留7天
  - 数据体量：单账号达到10万条交易记录时提示告警
- `[ ]` 公网访问（首发倾向：Cloudflare Tunnel）
  - 不要求路由器端口映射
  - 域名与证书由 Cloudflare 提供
  - DNS 托管（v1.3首发）：域名 Nameserver 迁移到 Cloudflare

## 3. 验收标准（可观察、可复现）
- `1.` 在设备A登录后新增一条记录，设备B在登录同账号后可自动同步看到该记录（无需手动点击）。
- `2.` 设备A编辑该记录金额/备注后，设备B在合理时间内自动同步到最新值。
- `3.` 设备A删除该记录后，设备B自动同步后该记录进入“回收站”或不在正常列表展示。
- `4.` 设备A从回收站恢复该记录后，设备B自动同步后记录恢复可见。
- `5.` 断网状态下设备A新增记录不会报错；恢复网络后自动补偿同步到服务器；设备B随后可同步到该记录。
- `6.` 公网通过域名 HTTPS 可访问；非 HTTPS 访问被拒绝或自动跳转。
- `7.` 登录接口不可暴露明文密码；密码在服务端以不可逆方式存储。
- `8.` 容器重启后数据不丢失（同一账号的数据仍可同步拉取）。
- `9.` NAS 资源影响可控：同步失败会指数退避，重试间隔上限为5分钟；不会持续高频请求；日志按天滚动且保留7天不无限增长。
- `10.` 邀请注册：无邀请码或邀请码无效/过期时无法注册；邀请码使用一次后作废；有效期为7天。
- `11.` 邀请码管理页：登录后可生成邀请码、查看其状态（有效/已使用/已过期）、并可作废未使用邀请码；未登录不可访问该页与对应接口。
- `12.` 登录态安全：登录成功后服务端下发 Cookie，包含 HttpOnly 与 Secure 属性；前端 JS 无法读取 token。
- `13.` 同步可观测性：设置页可看到同步状态（已同步/同步中/失败/离线）；点击“立即同步”可触发一次同步并反馈结果。
- `14.` 回收站清空：清空回收站需二次确认；确认后回收站内记录被永久删除且不可恢复；其他设备同步后同样不再可见/可恢复。
- `15.` 数据体量告警：当单账号交易记录达到10万条时，设置页提示性能与空间风险，并给出处理建议（如清理回收站/减少历史数据/导出备份）。
- `16.` 备份方案交付：需求文档内提供“如何备份与恢复”的可执行步骤（至少包含：数据落盘位置/备份对象/恢复步骤）。

## 4. 范围边界
### In（包含）
- 简单登录（注册/登录/退出）与账号隔离（单账号起步）
- 后端数据存储与基础 CRUD 能力
- 自动后台同步（含离线补偿）
- 冲突策略：最后写入覆盖
- 群晖 Docker 部署 + Cloudflare Tunnel 公网访问
 - 邀请注册的简易管理页（生成/查看/作废邀请码）
 - 资源约束默认值（同步退避、日志滚动、数据量告警）

### Out（不包含）
- 家庭成员/复杂权限模型（共享/私有账本、角色管理）
- 复杂冲突合并（按字段合并、手动选择）
- 复杂统计报表后端化
- 支付/订阅、第三方登录
 - 应用内一键导入/导出（仅提供运维级备份方案）

## 5. 约束与假设
- 前端仍为 React + TypeScript；现有数据结构为基础（Ledger/Category/Transaction/Template/Tag）。
- v1.3 首发优先保证“同步正确性与可部署”，再逐步优化体验与性能。
- 公网部署以“尽量降低 NAS 暴露面”为原则（优先 Cloudflare Tunnel）。

## 6. 风险与回滚
- 风险：
  - 自动后台同步若处理不当，可能导致数据覆盖与误删风险
  - 公网暴露带来安全风险（暴力破解、漏洞利用）
  - 多端时钟不一致会影响 LWW 决策
- 回滚：
  - 提供“切回纯本地模式”的开关策略（实现阶段定案）
  - 保留 v1.2 前端存档与可运行版本（已存档）

## 8. 方案制定（强制）
### 方案A：Node + SQLite（TS 全栈统一）
- 方案名称：Node（TypeScript）后端 + SQLite（WAL）+ Docker + Cloudflare Tunnel
- 核心思路：
  - 后端提供认证（邀请码注册/登录/登出）与增量同步（oplog 拉取/上报）API
  - 数据库存 SQLite，使用 WAL 提升并发读写表现；oplog 保留90天并定期截断
  - 登录态使用 HttpOnly Cookie；前端设置页显示同步状态并提供“立即同步”
- 技术选型：
  - 后端语言/框架：Node.js + TypeScript（框架可选 Fastify/Express）
  - 存储：SQLite（文件落盘到 NAS 卷）
  - 部署：docker compose（后端容器 + cloudflared 容器）
- 优点：
  - 与现有前端 TypeScript 语言一致，代码风格统一
  - SQLite 运维轻，适合单账号与 NAS 资源受限场景
- 缺点：
  - Cookie/CSRF 等安全细节需要严格实现，否则容易留下口子
  - Node 生态选择多，初期决策点较多（ORM/迁移/鉴权中间件等）
- 风险与应对：
  - 风险：SQLite 写入竞争导致性能抖动；应对：WAL + 合理事务粒度 + 同步退避
  - 风险：鉴权遗漏导致越权；应对：统一鉴权中间件 + 端到端验收用例覆盖

### 方案B：Python + SQLite（已选，推荐落地）
- 方案名称：FastAPI 后端 + SQLite（WAL）+ Docker + Cloudflare Tunnel
- 核心思路：
  - FastAPI 提供清晰的 REST API：认证、邀请码管理页 API、同步 oplog API、回收站 API
  - SQLite 作为主库：记录业务表 + oplog 表；oplog 保留90天并定期截断
  - 登录态使用 HttpOnly Cookie（服务端会话或签名票据均可），并配最小 CSRF 防护
  - 同步策略：事件触发 + 指数退避；设置页展示同步状态与“立即同步”
- 技术选型：
  - 后端语言/框架：Python + FastAPI（ASGI，运行时可选 Uvicorn）
  - 存储：SQLite（文件落盘到 NAS 卷）
  - 部署：docker compose（后端容器 + cloudflared 容器）
- 优点：
  - 接口声明清晰，调试与文档化友好（适合边做边教）
  - SQLite 运维轻，对 NAS 资源影响更可控
- 缺点：
  - Python 依赖管理需要规范（requirements/lock），避免“能跑但不可复现”
  - SQLite 仍不适合高并发写入，但 v1.3 单账号可接受
- 风险与应对：
  - 风险：同步/回收站/清空等操作的幂等一致性复杂；应对：oplog 设计先行 + 幂等键 + 事务包裹
  - 风险：Cookie 会话与 CSRF 配置错误；应对：默认 SameSite + 明确跨域策略 + 验收用例验证

### 方案C：Postgres + 后端（扩展性优先）
- 方案名称：Postgres +（Node 或 Python）后端 + Docker + Cloudflare Tunnel
- 核心思路：
  - 用 Postgres 解决并发与数据规模问题，并把 oplog 与业务表放在同一数据库体系内
  - 同步与鉴权逻辑与方案A/B一致，但运维与资源成本更高
- 技术选型：
  - 存储：Postgres（额外容器 + 数据卷 + 备份/升级成本）
  - 后端：Node 或 Python（取决于团队偏好）
- 优点：
  - 数据一致性、并发与扩展性更好；后续做多账号/家庭成员更顺滑
- 缺点：
  - NAS 资源占用更高，运维更复杂（备份、升级、监控）
- 风险与应对：
  - 风险：运维复杂度影响可用性；应对：先用 SQLite 跑通 v1.3，再评估迁移
