# 运维与备份（v1.3）

## 目标
- SQLite 数据可安全备份
- 故障时能恢复到最近一次备份
- 升级版本不会误删数据

## 数据与日志位置
默认（compose 具名卷）：
- 数据库：容器内 `/data/accounting.sqlite3`
- 日志：容器内 `/logs/app.log`（按天滚动，默认保留 7 天）

## 推荐备份策略
建议每天做一次备份，至少保留 7～30 份。

备份时优先保证“数据库文件 + WAL/SHM 一起一致”：
- `accounting.sqlite3`
- `accounting.sqlite3-wal`
- `accounting.sqlite3-shm`

### 方案 A（推荐，最简单可靠）：停后端再拷贝
1. 在群晖 Container Manager 停止 `accounting-backend`（保持 `cloudflared` 也一起停更简单）
2. 把数据卷目录下的 `accounting.sqlite3*` 复制到备份目录（按日期命名）
3. 再启动项目

优点：不会遇到 WAL 读写时机问题。

### 方案 B（不中断服务）：先做整库复制再备份
如果你希望不停止服务，建议用“整文件复制”并连带 `-wal/-shm`，并在备份完成后做一次校验：
- 校验方式：恢复到临时路径后启动一次后端，访问 `/health` 与登录流程

## 恢复流程（从备份恢复）
1. 停止项目
2. 用备份文件覆盖数据卷里的 `accounting.sqlite3*`
3. 启动项目
4. 登录后点击“立即同步”（会自动拉取基线/增量，修复多端状态）

## 升级流程（版本更新）
1. 先做一次备份（优先“方案 A”）
2. 更新代码或镜像
3. 重启项目
4. 观察日志 `/logs/app.log`：
   - `login_ok / register_ok`
   - `sync_push_ok / sync_compact_ok`
   - 是否出现异常堆栈

## 日志轮转
后端默认：
- 每天 0 点滚动
- 保留天数：7 天（可通过 `ACCOUNTING_LOG_RETENTION_DAYS` 调整）
- 目录：`ACCOUNTING_LOG_DIR`（默认 `./logs`，容器里建议挂到 `/logs`）
